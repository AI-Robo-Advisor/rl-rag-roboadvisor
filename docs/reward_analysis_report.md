## 1. 개요

`PortfolioEnv`(`src/rl/env.py`)에 구현된 보상 함수 3종(`return`, `sharpe`, `mdd`)을 각각 독립적으로 PPO 에이전트에 적용하여 Walk-Forward 방식으로 학습시켰다. 이 문서는 **각 보상 함수의 정의, 설계 의도, 학습 중 보상 신호 특성, 그리고 최종 선택 근거**를 RL 관점에서 기술한다.

백테스트 성과 지표(샤프비율, MDD, 누적수익률 등)는 `src/rl/backtest.py`(강유영 담당)가 생성한 결과를 참조하며, 이 문서에서는 해당 수치를 **보상 함수 거동 해석 목적으로만** 인용한다.

## 2. Reward Function 정의

보상 함수는 `PortfolioEnv.__init__`의 `reward_type` 인자로 지정하며, `_calculate_reward(net_return, weights)`에서 step마다 계산된다 (`env.py:136`).

공통 입력:
- `net_return`: 거래비용(fee 0.015% + 슬리피지 0.05% = 합계 0.065%)을 차감한 일별 포트폴리오 로그수익률
- `weights`: 현재 step에 적용된 포트폴리오 비중 (softmax 정규화 후)

### 2.1 Return Reward

```python
# env.py:149-150
if self.reward_type == "return":
    return net_return
```

**정의**: 거래비용 차감 후 일별 포트폴리오 로그수익률을 그대로 보상으로 반환한다.

**설계 의도**: 에이전트가 단기 수익 극대화에 집중하도록 유도한다. 가장 단순한 형태의 보상으로, 보상 스케일이 자산 수익률 크기에 직접 종속된다.

**특성**:
- 보상 범위: 시장 일별 수익률 수준 (대략 −3% ~ +6%)
- 위험 패널티 없음 — 변동성이 큰 포지션도 수익이 나면 동일하게 보상
- 시장 하락일에 큰 음의 보상이 발생하여 학습 신호 분산이 크다

### 2.2 Sharpe Reward

```python
# env.py:152-164
if self.reward_type == "sharpe":
    start = max(0, self.current_step - self.volatility_window)
    recent_returns = self.returns_df.iloc[start:self.current_step].values
    portfolio_returns = recent_returns @ weights
    volatility = float(np.std(portfolio_returns))
    if volatility < 1e-8:
        return 0.0
    return net_return / volatility
```

**정의**: 현재 step의 `net_return`을 직전 `volatility_window`(기본값 20 거래일) 구간의 포트폴리오 수익률 표준편차로 나눈 값이다.

> 주의: 이 값은 **step-level 보상 스케일의 샤프 비율**로, `metrics.py`가 계산하는 연율화 샤프 비율과 정의 및 해석이 다르다.

**설계 의도**: 수익률을 변동성으로 정규화하여 에이전트가 수익·위험을 동시에 고려하도록 한다. 동일한 수익이라도 변동성이 낮을수록 더 높은 보상을 받는다.

**특성**:
- 보상이 변동성 단위로 정규화되므로 학습 초기의 보상 스케일이 안정적
- 분모(변동성)가 0에 가까우면 보상 0을 반환하여 수치 불안정 방지
- 에이전트가 암묵적으로 저변동성 포지션을 선호하도록 유도

### 2.3 MDD Reward

```python
# env.py:166-167
if self.reward_type == "mdd":
    return net_return - self.lambda_mdd * self.current_mdd
```

**정의**: 일별 `net_return`에서 현재 시점의 최대낙폭(`current_mdd`)에 `lambda_mdd`(기본값 1.0)를 곱한 패널티를 차감한다.

`current_mdd`는 step마다 갱신된다 (`env.py:235-237`):

```python
self.current_mdd = (
    self.peak_portfolio_value - self.portfolio_value
) / self.peak_portfolio_value
```

**설계 의도**: MDD가 커질수록 보상이 감소하여 에이전트가 낙폭을 억제하는 행동을 선호하도록 유도한다.

**특성**:
- MDD 패널티는 낙폭이 **이미 발생한 뒤** 작동하는 후행(lagged) 신호
- 연속 하락 구간에서 `current_mdd`가 누적되어 보상이 지속적으로 음수로 유지됨
- `lambda_mdd = 1.0`이면 MDD 1%당 보상 −0.01의 추가 페널티가 부과됨

## 6. Reward별 관찰 결과

### 6-1. `return` 보상

- 보상 신호가 단순하고 직관적이나, 학습 중 **보상 분산이 크다**. 하락장 step에서 큰 음의 보상이 집중되어 PPO의 정책 업데이트 방향이 불안정해지는 경향이 있다.
- W1(금리 인상 하락장)에서도 양의 누적 수익(+55.4%)을 유지하여 기본적인 방향성 포착 능력은 검증되었다.
- 위험 항목을 보상에 반영하지 않으므로, 에이전트가 수익이 나는 한 변동성 높은 포지션을 동등하게 선호한다.
- W2~W3에서 MDD가 2~3%대로 우수한 것은 수익 추구 과정에서 자연적으로 분산 포트폴리오를 학습한 결과로 해석된다.

### 6-2. `sharpe` 보상

- `net_return / volatility`로 정규화된 보상은 **step 간 보상 크기가 안정적**이어서 PPO 학습이 수렴하기 유리한 환경을 제공한다.
- 에피소드 수(823)는 `return`과 동일하여 Safe-Guard 발동 빈도가 유사하고, 평균 보상이 3종 중 가장 높다(0.00403).
- 에이전트가 보상을 극대화하면 자연스럽게 **수익 극대화 + 변동성 최소화**를 동시에 달성한다 — `mdd` 보상 없이도 MDD가 낮게 유지되는 이유다.
- `sharpe` 에이전트의 베타가 윈도우가 지날수록 0.530 → 0.442 → 0.340 → 0.293으로 단조 감소 — 시장 노이즈를 점진적으로 걸러내는 방향으로 정책이 수렴하고 있음을 보여준다.
- 4개 구간 전체에서 누적 수익률·샤프 비율·알파가 최상위이며, MDD도 `mdd` 보상보다 낮다.

### 6-3. `mdd` 보상

- MDD 패널티는 **낙폭이 발생한 이후** 작동하는 후행 신호이다. 에이전트는 낙폭을 사전에 예방하는 법을 학습하기 어렵고, 대신 낙폭 발생 후 포지션을 축소하는 사후 회피 행동을 학습하는 경향이 나타났다.
- 연속 하락 구간(W1, 금리 인상기)에서 `current_mdd`가 누적되면서 보상이 지속적으로 음수로 유지된다. 이 환경에서 에이전트는 포지션 자체를 과소 취하는 방향으로 수렴하여 W1 누적 수익률 −10.6%, 샤프 −1.04를 기록했다.
- 에피소드 수가 758로 다른 보상(823)보다 65개 적다 — Safe-Guard(MDD 15%)를 더 자주 초과하여 에피소드가 조기 종료된 결과이며, MDD 억제라는 설계 목적과 반대되는 결과다.
- MDD 패널티가 실효적이려면 롤링 CVaR, 사전 변동성 신호 등 선행(leading) 위험 지표와 결합하거나 `lambda_mdd` 스케일링 및 보상 클리핑 범위 조정이 필요하다.

## 7. Safe-Guard Threshold 분석

`PortfolioEnv`는 `safe_guard_mdd = 0.15`(15%)를 초과하면 에피소드를 즉시 종료한다 (`env.py:59, 245`):

```python
safe_guard_triggered = bool(self.current_mdd > self.safe_guard_mdd)
terminated = bool(reached_end or safe_guard_triggered)
```

Safe-Guard가 보상 학습에 미치는 영향:

| 측면 | 내용 |
|------|------|
| 에피소드 조기 종료 | MDD > 15% 도달 시 즉시 종료 → 이후 누적 손실에 대한 보상 정보가 차단됨 |
| 학습 신호 기능 | 에이전트가 위기 구간에서 Safe-Guard 발동을 반복 경험하며 고위험 포지션을 회피하는 정책을 간접 학습 |
| `mdd` 보상과의 상호작용 | `mdd` 에이전트는 MDD 패널티 + Safe-Guard 이중 조기 종료 압력을 받아 에피소드 수가 가장 적게 기록됨 |

임계값 15%의 근거 및 적정성 검토 결과는 `docs/safeguard_threshold_analysis.md`에 상세히 기술되어 있다. 핵심 결론: **코로나 폭락 기간(동일가중 MDD 24.79%)과 금리 인상기(24.94%)의 MDD가 사실상 동일**하여, 임계값을 낮추지 않고 15%를 유지하는 것이 조기 경보 목적에 부합한다.

## 8. 최종 Reward 선택

**채택: `sharpe` 보상 함수**

| 판단 기준 | 내용 |
|----------|------|
| 학습 안정성 | 보상이 변동성 단위로 정규화되어 PPO 업데이트 방향이 안정적 |
| 다중 목표 내재화 | 수익 극대화 + 변동성 최소화를 보상 구조 자체에서 달성 |
| OOS 성과 일관성 | 4개 테스트 구간 전체에서 누적 수익률·샤프비율·알파 최상위 |
| Safe-Guard 적합성 | 에피소드 수(823)가 `return`과 동일 — MDD 임계값 초과 빈도가 `mdd` 보상보다 낮음 |
| 베타 감소 추세 | 윈도우 진행에 따라 시장 의존도가 지속적으로 낮아지는 정책 수렴 확인 |

**`return` 보상은 2순위 대안**으로 보관한다. 단순하지만 W2~W3에서 MDD가 2~3%대로 우수하여, 리스크 제어보다 수익성 우선 전략이 요구될 때 활용 가능하다.

**`mdd` 보상은 현 구현으로 비권장**한다. 후행 신호 문제로 인해 설계 의도(낙폭 억제)와 반대 결과가 나타났으며, W1에서 누적 손실이 발생했다. 개선 방향으로는 롤링 CVaR 기반 선행 패널티, Asymmetric 보상 설계, `lambda_mdd` 동적 스케일링 등을 검토할 수 있다.
